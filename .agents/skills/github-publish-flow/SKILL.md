# Github Publish Flow

## Overview

Use this skill to turn local repository changes into the correct GitHub action for this project: commit, PR, merge, tag, or push. Follow the repository policies first, then choose the minimal safe action that matches the user's intent.

## Decision Flow

1. Read the repository policies before acting:
   - [AGENTS.md](../../../../AGENTS.md)
   - [docs/branch-policy.md](../../../../docs/branch-policy.md)
   - [docs/tag-release-policy.md](../../../../docs/tag-release-policy.md)
   - [docs/11-production-operations.md](../../../../docs/11-production-operations.md)
2. Identify the target:
   - `main` for backend, policy, pipeline, and release-note changes.
   - `admin-deploy` only as a generated distribution branch for the frontend.
3. Choose the action:
   - `commit` when changes are local only.
   - `push` when the branch should be published on GitHub.
   - `PR` when the change must be reviewed/merged into `main`.
   - `tag` when the commit is a stable release milestone on `main`.
4. Never edit `admin-deploy` manually.
5. Never create a tag from `admin-deploy`.

## Branch Rules

- `main`
  - Source branch for backend and repository policy changes.
  - The GitHub Actions workflow publishes `admin-deploy` when `admin/**` changes land on `main`.
- `admin-deploy`
  - Generated by `git subtree split --prefix=admin`.
  - Used by the Heroku admin app.
  - Treat as a deployment artifact, not a development branch.
- Working branches
  - Use `feat/*`, `fix/*`, `docs/*`, `design/*`.
  - Use `codex/*` only as a temporary agent branch.

## GitHub Actions Expectations

- Keep the main CI workflow off `admin-deploy`.
- Validate frontend changes in the publish workflow before generating `admin-deploy`.
- If a requested push would affect both backend and frontend, prefer `main` first and let the publish workflow update the frontend branch.

## Tags and Releases

- Tag only stable `main` commits.
- Use SemVer tags like `v0.1.0`.
- Create a release note file when the tag is a meaningful milestone.
- Do not tag `admin-deploy`.

## When to Stop and Ask

- If the user asks to push changes but the intended branch is ambiguous.
- If a PR would mix backend, frontend, and documentation in a way that reduces review quality.
- If a rename would affect Heroku app URLs, remotes, or CORS and the change is not already planned.

## Output Discipline

- State the exact branch, commit, and destination.
- Prefer one clear action at a time.
- If a tag is created, report the tag name and the commit it points to.
